## Core Concepts

### Notes in `Daily/`

| Filename format       | Purpose                                                 |
| --------------------- | ------------------------------------------------------- |
| `YYYY-MM-DD.md`       | Date-stamped note. Can contain tasks, content, or both. |
| `YYYY-MM-DD HH.mm.md` | Timestamped note. Can contain tasks, content, or both.  |
|                       |                                                         |

All files in `Daily/` are equivalent — both formats can hold tasks, meeting notes, thoughts, plans, or any combination.

### Topic Pages

Topics live in `People/`, `Projects/`, and `Ideas/`. Each topic page is stamped by the Regenerate Indexes script with the Topic Template — a `> [!info] Generated File` callout followed by a `dataviewjs` block. **No manual content should be added.**

The `dataviewjs` block calls `inline-backlinks.js`, which finds all notes that link to the topic and renders each one's full markdown content inline, wrapped in a `> [!example] [[Filename]]` callout. Notes are sorted alphabetically by vault-relative filepath.

To associate a note with a topic, include a `[[WikiLink]]` to the topic anywhere in the note body.

Topic Pages also have the tag `#_moc/topic` on the last line to make them easy to find with grep.

### Topic Folder Structure

Each topic category (e.g., `People/`, `Projects/`, `Ideas/`) has a folder-level index note named `_ FolderName.md`. Individual topics are plain notes inside that folder:

```
People/
├── _ People.md        ← folder index note (auto-generated)
└── John Doe.md        ← individual topic note
```

The `_ *.md` naming pattern is what the Regenerate Indexes script uses to identify index notes. Individual topic notes (e.g., `John Doe.md`) do not use the underscore prefix.

### Maps of Content (MOC)

A **Map of Content** is an Obsidian concept for a note that primarily contains links to other notes, acting as a navigational index or hub. MOCs create a link hierarchy that Obsidian's graph view uses to visualize structure — MOC notes appear as high-connectivity hubs with spokes out to their linked notes.

This vault implements MOC at two levels:

- **`Index.md`** — the top-level MOC. Links to all `_ *.md` folder index notes, making it the root of the graph hierarchy.
- **`_ FolderName.md` notes** (e.g., `_ People.md`) — mid-level MOCs. Each links to all individual topic notes in its folder.

All MOC index files are auto-generated by the `Regenerate Indexes.md` Templater script. Do not edit them manually. Only daily notes contain actual handwritten note content.

## Plugins & Their Roles

A design principle of this vault is to rely exclusively on native Obsidian features and a small set of nearly ubiquitous plugins. Customization is minimized, and where necessary, implemented with the simplest approach possible using only these four widely-used plugins.

| Plugin | Role |
|---|---|
| **Dataview** | Powers `inline-backlinks.js` via `dataviewjs` blocks. JavaScript queries must be enabled in Dataview settings. |
| **Templater** | Runs `Regenerate Indexes.md` at vault startup. |
| **Tasks** | Parses and renders task items across the vault. Used in `All Tasks.md` and daily notes. |
| **QuickAdd** | Provides capture shortcuts for new notes and tasks. See [QuickAdd Shortcuts](#quickadd-shortcuts) below. |

## Implementation Details

### `inline-backlinks.js`

**Location:** `Utils/Dataview/inline-backlinks.js`

**What it does:** Reads `metadataCache.resolvedLinks` to find all notes linking to the current file, then renders each backlinker's full raw markdown content inline using Obsidian's `MarkdownRenderer`. Output is read-only and re-renders on every open — no files are modified.

**Usage** (paste into any note or template):
~~~
```dataviewjs
await dv.view('Utils/Dataview/inline-backlinks', {obsidian})
```
~~~

**Behavior:**
- Sorts backlinkers by vault-relative filepath (alphabetical)
- Excludes notes whose basename starts with `_ ` (index notes)
- Wraps each backlinker in a callout header: `> [!example] [[Filename]]`
- Preserves all Obsidian rendering: links, checkboxes, tags, embeds, formatting

**Limitations:**
- Unresolved links (notes that don't exist yet) won't appear
- Content is not editable in place

### `Regenerate Indexes.md` (Templater script)

**Location:** `Utils/Templater/Regenerate Indexes.md`
**Trigger:** Runs automatically at Obsidian startup via Templater's "Run on startup" setting.

**Three jobs, in order:**

1. **Rebuild topic index notes (mid-level MOCs)** — Finds all `_ *.md` files. For each, writes a list of `[[WikiLinks]]` to every sibling `.md` file in the same folder (alphabetically sorted). Overwrites the file completely.

2. **Rebuild `Index.md` (top-level MOC)** — Writes a list of `[[WikiLinks]]` to all `_ *.md` index notes found above. Overwrites `Index.md` completely.

3. **Stamp empty topic notes** — Finds all `.md` files inside indexed folders (i.e., folders that contain a `_ *.md` note) that are completely empty. Overwrites them with the content of `Utils/Templates/Topic Template.md`.

> **Important:** `Index.md` and all `_ *.md` index notes are auto-generated. Do not edit them manually — changes will be overwritten at next startup.

### Topic Template

**Location:** `Utils/Templates/Topic Template.md`

Contains a `> [!info] Generated File` callout followed by the `dataviewjs` block that calls `inline-backlinks`. This is what gets stamped onto empty topic notes by the Regenerate Indexes script.

### Daily Template

**Location:** `Utils/Templates/Daily Template.md`

Applied to new daily notes (`YYYY-MM-DD.md`). Contains a `---` horizontal rule separator. The QuickAdd `New Task Today` action appends new tasks below this separator.

### `All Tasks.md`

**Location:** `_ Views/All Tasks.md`

Uses the Tasks plugin query syntax to aggregate all tasks across the vault, grouped by the date embedded in the source filename:

~~~
```tasks
group by function \
task.created.moment?.format("YYYY-MM-DD") \
?? task.file.filename.match(/\d{4}-\d{2}-\d{2}/)?.[0] \
?? task.file.filename
```
~~~

### QuickAdd Shortcuts

| Shortcut                 | Action                                                                                                                 |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **New Daily Subnote**    | Creates and opens a new timestamped note fragment (`YYYY-MM-DD HH.mm.md`).                                             |
| **New Task Here**        | Inserts a new task at the current cursor position.                                                                     |
| **New Task Today**       | Appends a new task to the bottom of today's `YYYY-MM-DD.md` daily note.                                                |
| **New Task Today (URI)** | Same as New Task Today, but triggered via an `obsidian://` URI — intended for use with a macOS Shortcut global hotkey. |

### Hotkeys

The three in-app QuickAdd actions (New Daily Subnote, New Task Here, New Task Today) are bound to key combos via Obsidian's built-in hotkey settings. Additionally, macOS Shortcuts provides global hotkeys for **New Daily Subnote** and **New Task Today**, allowing capture without Obsidian being in the foreground. The URI-based `New Task Today (URI)` action exists specifically to support the macOS Shortcut global hotkey for adding tasks.

## Tasks Workflow

This system is **not a project management tool**. Project management belongs in dedicated tools (e.g., Jira in a software engineering context). This vault is complementary to such tools — tasks here are quick, short-lived action items with small scope. Typical examples: follow-ups from a meeting that need to happen the same day or week, reminders to create a Jira ticket, or small errands that don't warrant a tracked work item elsewhere.

- Tasks can be created in any note in `Daily/`. The containing note's name (date & time) will be used to group and sort tasks in the All Tasks view, giving a chronological presentation.
- `_ Views/All Tasks.md` provides a unified view of all tasks across the vault.
- Use the QuickAdd `New Task Today` action to insert a task into today's `YYYY-MM-DD.md`, or write one manually below the `---` separator in any daily note.
- Use the QuickAdd `New Task Here` action to insert a task at the cursor position while composing any note.
- Task format follows the Tasks plugin convention:
  ```
  - [ ] Task description -- #action @ HH:mm ✚ YYYY-MM-DD
  ```
- Completed tasks are marked with `- [x]` and get a completion date appended automatically by the Tasks plugin.

## Sync & Backup

This vault uses **Obsidian Sync** to keep content synchronized across devices: work laptop, personal laptop, phone, and tablet.

On the personal laptop, the vault also has a **git repo stored as a sibling** — the working tree and `.git` directory are separated (`…/Work` and `…/Work.git` rather than `…/Work/.git`). This keeps the git repo out of Obsidian Sync, which would otherwise propagate git internals to every synced device.

Periodically, the git repo is updated for archival backup:

```bash
git add . && git commit -m update && git push
```

The remote is a **private & local home NAS git server** used purely for archival purposes. It is not accessible from the public internet, but rather only when the personal laptop is connected to the home network.

## Agent Rules & Guidelines

1. **New content goes in `Daily/`** — All new notes (meetings, thoughts, research, plans, tasks) go in a new file: `Daily/YYYY-MM-DD HH.mm.md` for timestamped content, or `Daily/YYYY-MM-DD.md` for the day's note. Both formats support tasks and free-form content.
2. **Linking** — Use `[[WikiLinks]]` to connect notes to relevant topics in `People/`, `Projects/`, or `Ideas/`. This is what makes topic pages work. Find existing Topic pages to link by grepping files for the `#_moc/topic` tag.
3. **Never edit topic pages** — `_ *.md` index notes and `Index.md` are auto-generated. Topic notes (e.g., `People/John Doe.md`) are stamped by the Regenerate Indexes script if empty — do not add manual content.
4. **Formatting** — Standard Markdown. No YAML frontmatter unless specifically needed.
5. **Agent instructions** — All agent instructions and technical documentation reside in this root `AGENTS.md` file. Claude Code does not natively use this file, so a redirect in `.claude/CLAUDE.md` enables it to find and use `AGENTS.md`.
6. **Memory** — Update this file under the following Shared Memory section when key decisions or long-term preferences are established.
